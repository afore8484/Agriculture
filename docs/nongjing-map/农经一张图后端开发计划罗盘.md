# 农经一张图后端开发计划罗盘

## 1. 文档定位

本文档用于约束“农经一张图”后端开发方向，防止后续执行中偏离模块范围、技术路线、接口口径和数据库约束。

本罗盘不是任务清单，而是执行时必须反复对照的“方向校验基线”。

## 2. 方向锚点

### 2.1 模块边界
- 只处理“农经一张图”模块
- 不扩展到其他模块
- 严格沿用 7 个二级专题
- GIS 作为横切能力，不单独做成独立业务主题

### 2.2 技术锚点
- 后端技术栈固定：Java 17+、RuoYi-Vue-Pro 体系、Spring Boot、MyBatis-Plus、Flyway、Maven
- 数据库固定：PostgreSQL 优先，设计上兼容金仓
- 项目形态固定：单体后端，不拆微服务
- 后端代码目录固定：`modules/nongjing-map/backend`

### 2.3 契约锚点
- 接口统一前缀固定为：`/api/nongjing`
- 当前标准接口数量基准：68 个
- 数据结构基准为统一分析读库，不重复建专题库
- GIS 字段在 v1 阶段按 `geom_wkt` / `centroid_wkt` 文本能力处理，不强依赖 PostGIS 空间函数

## 3. 基准文档优先级

执行中若出现理解冲突，按以下优先级处理：

1. `农经一张图-需求梳理.md`
   作用：决定“做不做、属于不属于本模块、页面和业务理解是否正确”
2. `农经一张图标准接口文档.md`
   作用：决定“后端要暴露哪些接口、路径是什么、字段口径是什么”
3. `农经一张图数据结构文档.md`
   作用：决定“表结构、字段来源、汇总口径、事实表粒度”

处理原则：
- 如果代码和文档冲突，以文档为准
- 如果三份文档之间冲突，不允许直接在代码里自行折中，必须先回到文档层修正

## 4. 不可偏离项

以下事项一旦偏离，必须立即停下校正：

### 4.1 范围偏离
- 新增了需求中没有的模块
- 把 GIS 单列为独立业务模块
- 把“农经一张图”做成多个重复子系统

### 4.2 技术偏离
- 绕开既定 Java 技术栈另起一套后端
- 为了图省事直接写成临时脚本服务
- 在数据库层强依赖 PostgreSQL 专有空间函数，导致金仓兼容性失控

### 4.3 接口偏离
- 使用 `/api/v1` 或其他非 `/api/nongjing` 路径
- 返回字段与接口文档口径不一致
- 明细、趋势、导出、钻取接口只做了一半

### 4.4 数据偏离
- 重复建表
- 忽略统一维度表，导致专题各自维护区划、组织、时间口径
- 为了接口方便，随意改事实表粒度

## 5. 决策守则

后续开发过程中，所有技术决策都必须服从以下四条：

1. 先保证需求口径正确，再追求实现速度
2. 先保证接口与数据结构闭合，再追求页面联调快感
3. 先保证 PostgreSQL/金仓兼容，再考虑 PostgreSQL 特有优化
4. 先保证单体结构清晰，再考虑拆分和扩展

## 6. 偏差识别规则

执行中出现以下信号，视为已经偏离：
- 后端接口名称开始与文档不一致
- SQL 出现明显不可迁移到金仓的专有能力依赖
- 同一个页面指标，在接口、SQL、前端展示三处口径不一致
- 导出、趋势、明细只有页面需求，但后端无对应能力
- 总览页与专题页统计口径不一致

## 7. 偏差处理规则

一旦发现偏离，处理顺序固定如下：

1. 先标记偏离点属于哪一类：范围、技术、接口、数据
2. 回到三份基准文档定位冲突源
3. 先修正文档理解或实现方案，再继续编码
4. 禁止通过“临时兼容写法”把偏差带入下一阶段

## 8. 阶段校验口令

每个开发阶段结束前，必须逐条自检：
- 我现在做的内容是否仍然只属于“农经一张图”？
- 接口路径是否仍然全部是 `/api/nongjing`？
- 数据模型是否仍然共用统一维度和统一分析库？
- 当前实现是否仍然兼容 PostgreSQL/金仓约束？
- 如果今天交给别人继续做，对方是否会因为口径漂移而返工？

## 9. 最终使用方式

后续所有后端开发任务，都必须同时对照：
- 本罗盘，判断方向是否偏
- 执行计划，判断阶段是否正确
- 三份基准文档，判断内容是否正确

如果三者不一致，优先停下来校正，而不是继续开发。
