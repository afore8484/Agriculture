# 全接口回归测试基线（冻结）

更新时间：2026-03-22  
适用范围：`modules/village-finance/backend` + `ruoyi-vue-pro/yudao-server` 集成态

## 1. 基线目标

1. 固化“全接口可回归”的最小执行链路。
2. 明确接口回归分层，避免把“路由可达”与“业务正确”混为一谈。
3. 形成可重复执行的脚本与结果产物，作为后续每轮回归基线。

## 2. 回归分层

1. L1（自动）：路由可达性回归  
说明：验证接口在集成态下可访问，不是“请求地址不存在/静态资源不存在”。
2. L2（逐步自动化）：业务场景回归  
说明：验证正向/反向业务规则（参数、状态、事务、一致性、口径）。

## 3. 基线产物（已落地）

1. 接口目录（代码提取）  
`output/finance-api-catalog.csv`
2. 回归用例基线（每接口 3 条：L1/L2正/L2反）  
`output/finance-regression-cases.csv`
3. 路由回归结果  
`output/finance-route-regression-result.csv`  
`output/finance-route-regression-result.json`
4. 基线一致性校验结果  
`output/finance-regression-baseline-validation.json`

## 4. 固定执行命令

1. 生成接口目录  
`powershell -ExecutionPolicy Bypass -File scripts/generate_finance_api_catalog.ps1`
2. 生成回归用例基线  
`powershell -ExecutionPolicy Bypass -File scripts/generate_finance_regression_cases.ps1`
3. 执行全接口路由回归（L1）  
`powershell -ExecutionPolicy Bypass -File scripts/run_finance_route_regression.ps1`
4. 校验“接口目录 vs 用例基线”一致性  
`powershell -ExecutionPolicy Bypass -File scripts/validate_finance_regression_cases.ps1`

## 5. 首轮执行结论（2026-03-22）

1. L1 路由回归：`175/175` 通过。
2. 基线一致性：  
`catalog_endpoint_count=175`  
`case_count=525`（每接口 3 条）  
`missing_case_items=0`  
`extra_case_items=0`
3. L2 业务层当前仍需继续修复的已知问题：  
凭证相关查询在部分环境存在字段口径不一致（示例：`fin_voucher` 的 `id/book_id` 与现库字段命名不一致），会导致 `code=500`，需在“业务模型/数据结构”层收口后再执行全量 L2 自动化。

## 6. 后续回归规则（冻结）

1. 每次后端改动后至少执行一次 L1 全接口回归。
2. 涉及业务规则变更时，必须补充对应 L2 场景用例并记录结果。
3. 回归结果必须落盘到 `output/`，并在 `docs/village-finance/log/` 追加阶段日志。
4. 未经用户明确指令，不修改前端页面组件；仅调整 API 与测试基线。
