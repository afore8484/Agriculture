# S3-P0 统一“业务来源 -> 凭证关联模型”

- 时间：2026-03-18
- 阶段：S3-P0
- 范围：二阶段入口基础包（统一凭证来源模型）

## 1. 本轮范围

1. 仅实现统一“业务来源 -> 凭证关联模型”基础能力（P0）。
2. 接管现有内部转账制证入口到统一服务层。
3. 不展开资产/合同/审批/银农直联/党建业务功能实现。

## 2. 已完成事项

1. 数据结构：
   - 新增 Flyway：`V5__voucher_biz_link.sql`。
   - `fin_voucher` 新增 `biz_id` 字段与 `(voucher_type, biz_id)` 索引。
   - `fin_voucher_recycle` 新增 `biz_id`，确保回收还原不丢来源。
2. 统一服务层：
   - 新增 `BizVoucherLinkService`、`BizVoucherLinkServiceImpl`。
   - 新增 `BizVoucherCreateCmd`、`BizVoucherTraceRespVO`。
   - 统一能力：`createVoucher`、`traceByVoucherId`、`queryVoucherByBiz`。
3. 统一来源规则：
   - 新增 `FinanceVoucherType`，集中管理凭证来源类型。
   - 受管来源类型必须携带 `bizId`；`MANUAL` 不允许绑定 `bizId`。
4. 主线接管：
   - 内部转账制证改为 `FinanceFundsServiceImpl -> BizVoucherLinkService`。
   - 保持“转账入库 + 凭证生成 + voucherId 回写”同事务。
5. 一致性增强：
   - `VoucherServiceImpl` 转账凭证校验升级为 `voucher_type + biz_id` 联合校验并反查业务主表。
   - 回收站链路新增 `bizId` 保存与恢复。

## 4. 未完成事项

1. 资产、合同、审批、银农直联、党建模块尚未接入统一模型（待 S3 后续分包推进）。
2. “凭证来源反查接口”当前仅服务层能力，控制器接口未开放。

## 5. 风险提示

1. 历史数据若存在旧转账凭证无 `biz_id`，后续执行转账凭证写操作会被一致性校验拦截，需要数据修复策略。
2. 随二阶段模块接入增多，应补充跨模块并发场景下的来源唯一性与事务冲突回归用例。
